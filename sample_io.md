# Sample Q&A

These were produced by calling the rule-based router directly, which is deterministic and needs no API key, so anyone can reproduce them exactly:

```bash
python -c "from agent.rule_based_router import run_agent_rule_based as run; print(run('YOUR QUESTION'))"
```

When `GEMINI_API_KEY` is set, `run_agent()` answers the same questions through Gemini instead — the facts stay identical (since they come from the same tool calls) but the exact sentence wording is generated by the model each time, so it won't be word-for-word the same as below. A real Gemini-generated example is included at the end.

| # | Question | Agent's answer | Tool(s) called |
|---|---|---|---|
| 1 | `What is the status of ORD-1005?` | Your order ORD-1005 (Trail Blazer Hiking Boots (x1)) -- it was delivered on 2026-06-02. Tracking number: TRK-88190. | `get_order` |
| 2 | `Is there a cheaper alternative to what I ordered in ORD-1003?` | Yes! A cheaper alternative to WaveSound Bluetooth Headphones ($59.99) is EchoBeam Portable Speaker at $39.99 (in stock). | `get_order` → `get_product` → `search_products` |
| 3 | `Where is order ORD-1111?` | I couldn't find any order with ID ORD-1111. Could you double-check the order number? | `get_order` (returns nothing found) |
| 4 | `Do you have any apparel?` | Here are some options I found: CozyKnit Wool Sweater ($45.00); RainGuard Packable Jacket ($65.00). | `search_products` |
| 5 | `Tell me about PRD-101` | WaveSound Bluetooth Headphones is priced at $59.99 and is in stock. Over-ear wireless headphones with 30hr battery life. | `get_product` |
| 6 | `Tell me about PRD-555` | I couldn't find any product with ID PRD-555. Could you double-check the product ID? | `get_product` (returns nothing found) |
| 7 | `Do you have any laptops?` | I couldn't find any products matching that. Could you try different keywords? | `search_products` (empty result) |
| 8 | `hello there` | I'm not sure what you're asking about. Could you share an order number (e.g. ORD-1002), a product ID, or describe what product you're looking for? | none -- nothing to look up, so it asks instead of guessing |

## What each row is meant to demonstrate

- **Row 1** — the most basic case: one question, one tool call.
- **Row 2** — chaining: the agent has to look at what was ordered before it can search for something cheaper. This is the row that actually shows multi-step reasoning rather than a single lookup.
- **Row 3 & 6** — bad IDs don't crash the program or get a made-up answer; the tool returns nothing and the agent says so.
- **Row 4** — a category search returning multiple real results.
- **Row 7** — proves the agent won't invent a product just because none matched.
- **Row 8** — with literally nothing to go on, the agent asks rather than assumes.

## Real LLM output for comparison

Running row 2's question through Gemini (live, deployed version) instead of the rule-based router gave:

> The ClassicLeather Oxford Shoes you ordered in ORD-1001 cost $149.
> Here are some cheaper alternatives, though they are different styles of shoes:
> - UrbanStep Casual Sneakers: $54.99, great for everyday walking
> - Aero Runner Sneakers: $79.99, lightweight running shoes

(Different order ID used in this particular live test, but same behavior.) Note that Gemini decided to skip straight from `get_order` to `search_products`, without calling `get_product` in between — it reasoned that the order's item name was already enough to search with. That's the model choosing its own path, not a fixed script.
